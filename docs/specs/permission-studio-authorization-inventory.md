# Permission Studio authorization inventory

**Audit date:** 2026-07-22  
**Scope:** `apps/api/src` HTTP controllers, WebSockets, BullMQ workers, scheduled jobs,
downloads/shares, API-key paths, integrations, and platform/support operations.  
**Status:** research baseline; proposed permissions are not implemented.

## Reading the inventory

- Roles: `PSA` platform superadmin, `PS` platform support, `TO` tenant owner, `TA` tenant
  admin, `WA` workspace admin, `WM` workspace member, `PM` project manager.
- Current guards: `J` JWT only, `R(A)` workspace `RolesGuard` ADMIN, `PM` adds
  `AdminOrProjectManagerGuard`, `TR(...)` tenant role guard, `PG` platform guard (support or
  superadmin), `PSG` platform superadmin guard, `K` API-key guard, `P` public/token/webhook.
- Context is the resource that must be resolved from trusted data before authorization. URL/body
  identifiers are locators, not authorization evidence.
- Gap: `MISSING` means no canonical evaluator call; `COARSE` means role/authentication is checked
  but not the intended permission; `PARTIAL` means a service performs an ownership or scoped check;
  `PUBLIC` means intentionally unauthenticated and requires non-RBAC controls; `COVERED` is reserved
  for a route proven to call the canonical evaluator.
- No HTTP controller directly calls `AuthorizationPolicyService`. The only runtime evaluator callers
  found are role-grant policy (`tenant:ManageWorkspaceAdmins`, `workspace:ManageMembers`,
  `project:ManageTeam`) and project team access (`project:ManageTeam`).
- A `+` permission below is a proposed v2 ID absent from the current 46-ID v1 catalog.

  helpdesk controllers are placeholders with no methods.

  impersonation flow; process-local permission editor claiming success; actorless bulk workers;
  export/download execution reauthorization; support WebSocket joining every platform principal;
  and broad ADMIN/PM guards standing in for task, report, presence, and timesheet permissions.
