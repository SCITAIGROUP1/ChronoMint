2. What's different for the Admin app

The Login and Forgot Password screens are visually and structurally identical to the Client app (shared @kloqra/ui package) — so this pass focused on what actually differs for a Workspace Admin / Tenant Owner:

  • A Tenant Owner account with 2+ workspace memberships is routed to a new /select-context picker after login (not seen in the Member/client-app flow), before landing on a workspace dashboard.
  • The post-login dashboard nav is materially richer: Team Management, Project managers, Projects, Categories, Team Live, Approvals, Hourly rates (billing), Exports, Workspace settings — confirming AC-12's role-based access split.
  • Because the Reset Password form shares the same component as the client app, the KAN-8 double-submit regression tests were re-run here to check whether that bug is shared-component-wide — it is, on the Login form (see Section 5). A brand-new, more severe defect (KAN-9) was found specifically here.

Healing activity log

Issue found: the [KAN-8 cross-app check] Enter-then-click must not double-submit /auth/reset-password test timed out (45s) waiting to click "Reset password" a second time.

Root cause: not a flaky selector — a real interaction between the two bugs. KAN-9 redirects the Reset Password page to /login immediately after the first (Enter-triggered) submit, before the follow-up click can ever fire — so the button the test was waiting for no longer exists on the page.

Fix applied: the page object now checks whether the button is still visible before attempting the second click; the test detects a post-Enter redirect to /login and explicitly test.skip()s with an annotation explaining it's blocked by KAN-9, instead of crashing with a misleading timeout.

Result after fix: stable re-run — the skip is now a clear, intentional signal instead of noise, and the two genuine defect-proof failures (KAN-9 itself, and KAN-8 on the Login form) are unaffected and reproduce consistently.

6. Defect log (Step 6)

KAN-9 — Admin app: invalid/expired reset-password token silently redirects to /login instead of showing the error

Severity: Critical   Priority: High   Label: AI_Generated

Impact: completely breaks the password-recovery flow for every Admin user whose reset link has expired or is invalid — precisely the users this feature exists to help. Reproduced twice, including from a fully clean browser session (no cookies/localStorage), ruling out leftover state as a precondition.

Root cause: the Reset Password form's 401 is handled by the same "authenticated" HTTP client wrapper used for in-app data (401 → silent refresh → redirect-to-login-on-failure), instead of the public/unauthenticated wrapper Login and Forgot Password correctly use.
